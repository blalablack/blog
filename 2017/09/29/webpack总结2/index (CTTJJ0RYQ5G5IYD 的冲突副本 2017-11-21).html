webpack总结2

发表于 2017-09-29 | 分类于 Webpack

这篇主要是对webpack生产环境优化的总结，主要包括配置生产环境，代码压缩，代码分离，缓存这几部分内容。至于按需加载，测试代码等会在后续添加。

一、摘要

此部分主要是概括了一部分生成环境优化的过程，其实核心还是根据浏览器的特性进行优化。

  • 分离css文件，可以实现样式分离，做到css和js的并行加载，提高页面加载渲染加载速度。
  • 分离js（第三方库，本地js，manifest）实现缓存，每次修改用户只要更新修改的部分。
  • 按需加载，尤其单页面应用减少首次加载文件，提高速度。

二、代码分离

官方文档对这一部分做了详细的说明。

概述

webpack的代码分离能够把代码分离到不同的 bundle 中，然后可以按需加载或并行加载这些文件。代码分离可以用于获取更小的 bundle，以及控制资源加载优先级，如果使用合理，会极大减少加载时间。

方法

entry分离

这是最简单的分离方式，前面也说到过就是设置多个入口，打包成不同的文件。

缺点：如果入口 chunks 之间包含重复的模块，那些重复模块都会被引入到各个 bundle 中。例如你a.js和b.js同时引用了base.js，那么两个打包的bundle都会包括这个base.js，造成重复引用。

防止重复

CommonsChunkPlugin
为了解决上面提到的问题，我们可以使用CommonsChunkPlugin插件，它可以将公共(必须所有入口都有这个依赖)的依赖模块提取到已有的入口 chunk 中，或者提取到一个新生成的 chunk。就是把上面的base.js单独拿出来。这样就避免了重复引用的问题。

注意：CommonsChunkPlugin会提取webpack 的样板(boilerplate)和 manifest（更改任意的都会改变），那么提取的公共模块岂不是不能缓存了吗？我的个人想法是既然这个文件是公共的，你可以像处理第三方库一样多一个入口多生成一个新chunk，webpack 的样板(boilerplate)和 manifest单独放一起，这样就不会干扰缓存。

ExtractTextPlugin
ExtractTextPlugin插件会将所有的入口 chunk(entry chunks)中引用的 *.css，移动到独立分离的 CSS 文件（公共的就是生成单独文件名字是下面plugins定义的名字，否则就按照entry生成不同的css文件）。因此，你的样式将不再内嵌到 JS bundle 中，而是会放到一个单独的 CSS 文件（即 styles.css）当中。 如果你的样式文件大小较大，这会做更快提前加载，因为 CSS bundle 会跟 JS bundle 并行加载。

三、缓存

CommonsChunkPlugin

提取模板

CommonsChunkPlugin能够在每次修改后的构建结果中，将 webpack 的样板(boilerplate)和 manifest 提取出来。通过指定 entry 配置中未用到的名称，此插件会自动将我们需要的内容提取到单独的包中，一定要单独的包，因为任何的变动都会导致它的变化。

这里打包生成的runtime就是webpack 的样板(boilerplate)和 manifest，这样就可以利用缓存了。

关于manifest，大致可以理解为webpack用来管理实现模块的文件，每次任何文件改变打包后都会改变那么缓存就无从说起了，所以才有了用CommonsChunkPlugin提取manifest。
manifest说明

提取第三方库

第三方库（react，jq..）很少像本地源代码那样频繁修改，可以利用CommonsChunkPlugin把他们打包到一个单独的文件中，同时使用chunkhash，就可以达到实现缓存。

注意，引入顺序在这里很重要。CommonsChunkPlugin 的 vendor 实例，必须在 runtime 实例之前引入。
HashedModuleIdsPlugin用来标记模块，防止verdor等因为模块数目改变而改变。

执行构建会打包成三个文件vender、index、runtime。

  • vender 包含第三方库，基本不会改变，可以实现缓存。
  • index 本地自己写的js
  • runtime webpack管理文件
提取css

用上面提到的ExtractTextPlugin插件提取css文件，同时通过[contenthash]设置合理缓存。
